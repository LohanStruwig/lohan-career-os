# Public Visual Proof Guidelines

Visuals can help the portfolio feel more concrete, but they should be added only when they improve clarity without exposing private material. This repository should not use screenshots or diagrams as a substitute for unavailable proof.

## Visual Principles

- Keep visuals optional, restrained, and professional.
- Use visuals to clarify a public narrative, not to imply private evidence.
- Prefer simple diagrams, public website screenshots, or purpose-built public-safe images.
- Avoid decorative visuals that make the portfolio feel less credible.
- Do not add visuals that require paid services, metered storage, or external tracking.

## Safe Visual Candidates

Good candidates:

- public VentureLayer Digital website screenshots;
- simple workflow diagrams built from public-safe labels;
- sanitized product-flow diagrams for CNC Intel Hub;
- public-safe case-study summary cards;
- generic operating-model diagrams that do not expose private systems;
- GitHub repository screenshots that show public README, issues, PRs, or release history.

## Do Not Use

Do not publish visuals containing:

- credentials, tokens, API keys, or `.env` values;
- private customer, employer, partner, or contact data;
- names, emails, phone numbers, account identifiers, or addresses that are not intentionally public;
- private dashboards, analytics, admin panels, database screens, logs, or system access views;
- private source documents, raw exports, or unreviewed research files;
- compensation, tax, medical, legal, insurance, or role-application materials.

## Recommended First Visuals

If visuals are added later, start with the lowest-risk set:

- a public screenshot of `venturelayerdigital.com`;
- a simple VentureLayer operating-model diagram using public-safe labels;
- a sanitized CNC Intel Hub workflow diagram showing research intake, review, saved views, and PDF summary output;
- a GitHub workflow screenshot showing public PR/release discipline.

## Review Before Publishing

Before adding any visual:

- run the [Public Review Checklist](public-review-checklist.md);
- inspect the image at full size;
- confirm no hidden browser tabs, file paths, account names, or local system details are visible;
- confirm the visual still makes sense without private context;
- use a PR so the visual can be reviewed before it lands on `main`.

## Current Decision

No screenshots or visual assets are added by default in this pass. The next visual step should be a separate, reviewed implementation after choosing one low-risk public visual from the recommended set.
